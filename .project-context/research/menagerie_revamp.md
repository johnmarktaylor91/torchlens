# Model Menagerie Revamp -- Design Notes (Deferred)

> Status: DEFERRED. Captured 2026-04-27 from architect discussion.
> Owner: JMT. Not slotted into a sprint yet.

## Problem

Current "model menagerie" is a manually maintained Google Drive folder
(linked from README.md lines 7 and 61). JMT dislikes the manual upkeep.
Goal: auto-update the gallery from CI so visualizations stay in sync
with code, with no manual hosting step.

## What Already Exists (Don't Rebuild)

- `scripts/build_torchlens_theme_gallery.py` -- working 10-model gallery
  builder for the dagua theme. Generates PNGs + `manifest.json` +
  auto-generated README.md. This IS the menagerie pattern, just
  scoped narrowly. Use as the template.
- `tests/test_real_world_models.py` -- already calls
  `show_model_graph(..., vis_save_only=True, vis_outpath=...)` for
  ~100+ real-world models (alexnet, vgg16, vit, googlenet, etc.),
  writing to `VIS_OUTPUT_DIR`. Outputs are discarded after the test.
  These models are already validated to render correctly.
- CI (`tests.yml:38-41`) already installs graphviz on Ubuntu runners,
  so visualization works in CI today.
- `release.yml` runs on push to main via semantic-release, publishes
  to PyPI + GitHub Releases. Natural hook point for menagerie rebuild.

## Architectural Decisions (the real forks)

### 1. When to rebuild?

- Every push -> infeasible. Slow models take >5 min each, 700 models
  is way past GitHub-hosted runner limits.
- Release-tag only -> menagerie versioned with PyPI release. Cheap,
  predictable.
- Hybrid -> small smoke gallery on PR + full gallery on release.

**Recommendation: hybrid.** Smoke gallery (5-10 small models) on every
PR catches "did vis silently break?" cheaply. Full gallery only on
release.

### 2. Where to host?

- GitHub Pages on this repo, gh-pages branch -- zero new infra.
- Separate `torchlens-gallery` repo -- clean but more upkeep.
- External (Netlify/Cloudflare Pages) -- nicer previews but yet
  another account.

**Recommendation: GitHub Pages, gh-pages branch.** URL:
`johnmarktaylor91.github.io/torchlens`. Replace Google Drive link in
README.

### 3. Where to store the heavy artifacts?

This is the underestimated decision. Graphviz PDFs of swin/vit-style
models are 10-50MB. ~100 models * 20MB = 2GB. That doesn't fit Pages
cleanly.

- **Option A (recommended): PNG thumbnails on Pages, full PDFs as
  GitHub Release assets, linked from each entry.** Pages stays small
  (~100MB), PDFs versioned with the release, free.
- Option B: Git LFS on gh-pages -- works but burns LFS quota.
- Option C: Cloudflare R2 / S3 -- best CDN, costs money.

### 4. Model coverage scope?

- 700+ from the paper -> infeasible on free GitHub runners.
- Curated 50-100 -> feasible, covers all architectural categories.
- Tiered: 20 always, 100 weekly cron, 700 nightly self-hosted.

**Recommendation: start ~50, organized by architecture family.** Most
models don't need pretrained weights for visualization
(`test_real_world_models.py` already instantiates with random init).

## Recommended Architecture

```
scripts/build_menagerie.py        # source of truth for specs + render
  -> tests/test_outputs/menagerie/
       index.html + per-category/index.html
       <model>/thumb.png  (small, ~200KB each)
       <model>/full.pdf   (uploaded to release; link only on Pages)
       manifest.json      (drives the index, machine-readable)

.github/workflows/menagerie.yml   # new workflow
  triggers:
    - on release published         (rebuild full + upload PDFs to release + deploy Pages)
    - workflow_dispatch            (manual)
    - on PR (smoke gallery only, 5-10 small models, fast)
  jobs:
    - matrix split across N runners by category
    - aggregate manifests
    - deploy via peaceiris/actions-gh-pages

README.md                          # update Google Drive links to Pages URL
```

Build script extends the existing `ReferenceSpec` dataclass pattern
from `build_torchlens_theme_gallery.py`. Same manifest shape, same
auto-generated README. Just expanded to many models, rendering with
graphviz (default) instead of dagua.

Index page: single static HTML generated by the script, no JS
framework. Filter/search via vanilla JS reading `manifest.json`. If
fancier filtering wanted later, swap in Astro -- but v1 doesn't need
it.

## Phased Rollout

1. **Phase 1 (1-2 days, mostly Codex):** generalize
   `build_menagerie.py` from the dagua theme gallery. ~20 known models.
   Generate locally, confirm output.
2. **Phase 2 (1 day):** `menagerie.yml` on `workflow_dispatch` only.
   First push to gh-pages. Verify Pages serves correctly.
3. **Phase 3 (1 day):** wire to release event. Add release-asset
   upload for PDFs. Update README links.
4. **Phase 4 (ongoing):** expand model coverage, filters/search,
   optional graphviz/dagua side-by-side toggle.

## Open Questions for JMT (when picking this up)

1. **Curated subset vs all 700?** Curated is realistic on free
   runners. 700 needs self-hosted runner.
2. **Graphviz only, or graphviz + dagua side-by-side?** Side-by-side
   forces dagua to stay current but doubles render cost. Suggest
   graphviz-only for v1, dagua toggle in phase 4.
3. **Multiple views per model (unrolled, rolled, depth-1..N) or one
   canonical view?** Existing test_alexnet generates 5 views. Nice
   but multiplies storage. Suggest one canonical + "show more views"
   link.
4. **Model list for v1?** Mirroring `test_real_world_models.py` is the
   obvious starting point since those are already validated.

## Risks to Flag

- **Curated lists rot.** Every new torchvision/timm/HuggingFace
  release adds models that should be in the menagerie. Worth deciding
  early: hand-maintained canonical set, or auto-iterated ("all
  torchvision classification models")? Suggest hybrid -- curated
  "canonical" tab + auto-iterated "torchvision/timm coverage" tab.
- **PDF size for large transformers.** Already addressed by the
  release-asset storage plan, but worth a sanity-check render on swin
  before committing.
- **CI minutes budget.** GitHub-hosted runners on a public repo are
  free but capped. Full menagerie rebuild + release publish + PyPI
  upload all in one workflow needs to stay under 6h.
